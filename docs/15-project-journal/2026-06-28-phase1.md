## Where We Started

## The First Decision: Spring Boot 4.1

The first meaningful fork in the road came before any scaffolding was written. Spring Boot 3.x is the stable, battle-tested choice. Spring Boot 4.1 is the frontier — newer, less documented, with breaking changes from the ecosystem it inherits.

We chose 4.1. The reasoning was deliberate: this is a learning project, and learning means encountering real engineering problems, not just solved ones. That choice paid off almost immediately.

Spring Boot 4.1 brought three breaking changes that weren't clearly documented anywhere we could find:

1. **Jackson 3.x** — the package root moved from `com.fasterxml.jackson` to `tools.jackson`. Every ObjectMapper import in security handlers silently failed to compile until we traced the dependency tree and found the renamed package.

2. **UserDetailsServiceAutoConfiguration** moved — the exclusion path that every Spring Security tutorial shows (`org.springframework.boot.autoconfigure.security.servlet`) no longer existed. We had to locate the jar manually to find the new home: `org.springframework.boot.security.autoconfigure`.

3. **Security auto-configuration split** — moved to a separate `spring-boot-security` jar rather than living inside `spring-boot-autoconfigure`.

None of these were fatal. All three were solvable within the session. But they were a reminder that choosing the frontier means being the person who documents the map.

## Building the Multi-Module Structure

The project is two microservices — `auth-service` and `engagement-service` — with shared infrastructure code that neither should duplicate. The Maven multi-module parent was the first structural decision.

The key insight here is that Maven multi-module isn't just about build organization. It's about enforcing dependency contracts. The parent `pom.xml` owns all version declarations through `<dependencyManagement>`. Child modules declare what they need; the parent ensures consistency. No service can accidentally pull in a different JJWT version than another.

The `common` module emerged from a principle Ravi identified early: duplication is the first sign of a boundary problem. When exception handling and JWT infrastructure were initially being written into both services, the right move was to pause, extract, and define a shared library. Common isn't a Spring Boot application — it's a plain jar with no `spring-boot-maven-plugin`. This distinction matters: it means common can be compiled, tested, and depended upon without trying to run as a standalone service.

The decision to instantiate common's classes as `@Bean` in each service's `SecurityConfig` rather than annotating them with `@Component` was deliberate library design. A library shouldn't impose component scanning behavior on its consumers. The service decides when and how to wire the beans.

## JWT and Security

The authentication infrastructure followed a clean layered model:

- `JwtProperties` — configuration bound via `@ConfigurationProperties`, immutable record
- `JwtService` — pure Java, no Spring annotations, fully testable in isolation
- `JwtAuthenticationFilter` — the Spring Security integration point, reads the Bearer token and populates the SecurityContext
- Entry point and access denied handler — return structured JSON errors (not Spring's default HTML error pages)

One non-obvious choice: `UserPrincipal` is a plain record, not a Spring Security `UserDetails`. We don't need the full `UserDetails` contract — we only need userId and role for authorization decisions downstream. Carrying the minimal principal keeps the security context lean and avoids tying domain concepts to framework interfaces.

The JWT itself carries three claims: `sub` (userId), `role` (MERCHANT or CUSTOMER), and `userType` (mirrors role — for forward compatibility if the two ever diverge). Symmetric key signing with HMAC-SHA256 was chosen over asymmetric RSA because all services share the same secret — there's no external token consumer that needs to verify without the secret. RSA would add complexity without adding security in this topology.

## Database Design and Flyway

Two databases, two services, no shared schema. This is the database-per-service pattern in practice, and its most important implication isn't the deployment topology — it's the absence of cross-database foreign keys.

`merchant_id` and `customer_id` appear in engagement_db tables, but they carry no FK constraints back to auth_db. The database cannot enforce referential integrity across service boundaries. That enforcement moves to the application layer: JWTs carry the userId, and the application trusts the JWT. This is a conscious trade — you gain service autonomy and independent deployability, and you accept that data consistency is an application responsibility.

Flyway was the natural fit for schema management. `ddl-auto=none` tells Hibernate to stay out of the DDL business entirely. Flyway owns schema creation and evolution. The checksum mechanism means every migration is immutable once applied — if you need to fix something, you add a new migration, you never edit an existing one. This is the discipline that makes database changes safe and auditable.

The seed migration for `feedback_topic_master` deserves a note. Rather than requiring merchants to manually configure topics after registration, the platform pre-loads master topics for each category. A restaurant gets Food, Service, Billing, Cleanliness, and General — out of the box, on day one. This is a product decision that the data model enforces: merchants don't create topics, they enable or disable from the master list. It keeps quality consistent and onboarding instant.

## The Collaboration Model

One thing that evolved mid-phase: the documentation agent that was supposed to maintain the docs folder failed. Rather than leaving documentation to drift from implementation, we absorbed that responsibility — implementation and documentation became one concern, not two. Every commit now carries both.

This turned out to be the right model anyway. The person (or agent) writing the code is best positioned to document the decisions behind it. Documentation that's written after the fact, by a different hand, tends to describe what was built rather than why. The why is what ages well.

## What Phase 1 Established

By the end of Phase 1, the project had:

- A Maven multi-module build with version governance at the parent level
- Two runnable Spring Boot services with full security filter chains
- A shared library with exception handling and JWT infrastructure
- Two PostgreSQL databases running in Docker with health checks
- Nine Flyway migrations covering the complete data model
- 24 seed rows of master feedback topics across five merchant categories
- Full documentation coverage: data dictionary, migration strategy, ER diagram, index strategy

The foundation is not glamorous. Nobody sees the Maven parent pom or the Flyway migration history from the outside. But it's the difference between a codebase that scales and one that collapses under its own weight at the first real feature.

## Looking Ahead

Phase 2 is where the project becomes visible. Registration endpoints, OTP challenge-response, JWT issuance — these are the flows a user actually touches. The foundation we built in Phase 1 exists to make Phase 2 straightforward: the exception handling is already wired, the security filter chain is already in place, the schema is already migrated. Phase 2 should feel like filling in a well-designed form rather than building the form while trying to fill it.

The one design decision already made for Phase 2: the Merchant Landing API will fetch merchant profile data from auth-service via REST rather than maintaining a denormalized copy in engagement_db. It's the simpler path for the MVP. If it becomes a bottleneck at scale, we'll revisit — but we'll revisit with real data, not speculation.
